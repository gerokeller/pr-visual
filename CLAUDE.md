## Releases

Releases are automated by `.github/workflows/release.yml`. The workflow runs after the `CI` workflow finishes successfully on `master`, so a red CI run blocks the release. Do not create tags or GitHub Releases by hand.

**Rules for AI agents:**

- **Default (patch bump): do nothing.** When your change is a fix, refactor, docs, chore, or any non-breaking / non-feature work, do not touch `package.json` or `.claude-plugin/*.json`. Merging to `master` will tag, release, and sync the version across all manifests automatically (e.g., `v1.1.3` â†’ `v1.1.4`).
- **Minor bump (new feature, backwards-compatible): bump `package.json` `version` to the next `MAJOR.MINOR.0`** (e.g., `1.1.3` â†’ `1.2.0`) in the same PR. The release workflow will pick up the bump, release that version, and propagate it to `.claude-plugin/plugin.json` and `.claude-plugin/marketplace.json` in a follow-up commit. You do not need to edit those two files by hand. Optionally add a new section to `CHANGELOG.md`.
- **Major bump (breaking change): bump `package.json` `version` to the next `MAJOR.0.0`** (e.g., `1.2.5` â†’ `2.0.0`). Same rule as minor. Breaking changes should be called out in the PR description and `CHANGELOG.md`.
- **Never commit a patch-only version bump.** The workflow handles those. Manual patch bumps will desync `package.json` from the tag sequence.
- **Manifest version fields are derived, not authored.** After each release, the workflow commits back a `chore(release): sync manifests to vX.Y.Z` change that updates `package.json`, `.claude-plugin/plugin.json`, and both `version` fields in `.claude-plugin/marketplace.json`. CI and CodeQL run on that commit (so branch protection is satisfied); the Release workflow detects its own commit subject and exits without tagging again. Do not open PRs that only touch these version fields.
- **Source of truth for the released version is the latest `v*` git tag.** `package.json` is consulted to detect manual minor/major bumps (i.e., when it is strictly greater than the latest tag).
- Release notes are auto-generated by GitHub from merged PRs / commits since the previous tag. Keep PR titles descriptive â€” they end up in the release notes verbatim.

**How the workflow decides:**

1. Trigger: `workflow_run` after `CI` completes on `master`. If CI failed, the release job is skipped. `workflow_dispatch` is also available for manual re-runs.
2. Compute `pkg` from `package.json` and `latest` from `git tag --list 'v*' --sort=-v:refname | head -1`.
3. If `pkg > latest` (SemVer-sorted): release `v$pkg` (manual minor/major).
4. Otherwise: release `v<latest with patch incremented>`.
5. If the resulting tag already exists, skip (idempotent re-run protection).
6. If `pkg` differs from the released version, rewrite the four manifest version fields, commit `chore(release): sync manifests to vX.Y.Z`, and push to `master`. That push triggers CI (and CodeQL) again, but the Release workflow self-skips on commits whose subject starts with `chore(release): sync manifests `, avoiding a feedback loop.
