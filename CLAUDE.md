# pr-visual — agent notes

## Releases

Releases are automated by `.github/workflows/release.yml` on push to `master`. Do not create tags or GitHub Releases by hand.

**Rules for AI agents:**

- **Default (patch bump): do nothing.** When your change is a fix, refactor, docs, chore, or any non-breaking / non-feature work, do not touch `package.json`. Merging to `master` will tag and release the next patch automatically (e.g., `v1.1.3` → `v1.1.4`).
- **Minor bump (new feature, backwards-compatible): bump `package.json` `version` to the next `MAJOR.MINOR.0`** (e.g., `1.1.3` → `1.2.0`) in the same PR. Also add a new section to `CHANGELOG.md` describing what changed.
- **Major bump (breaking change): bump `package.json` `version` to the next `MAJOR.0.0`** (e.g., `1.2.5` → `2.0.0`). Same rule for `CHANGELOG.md`. Breaking changes should be called out clearly in the PR description and changelog entry.
- **Never commit a patch-only version bump to `package.json`.** The workflow handles those. Manual patch bumps will desync `package.json` from the tag sequence.
- **Source of truth for the released version is the latest `v*` git tag**, not `package.json`. `package.json` is only consulted by the workflow to detect manual minor/major bumps (i.e., when it is strictly greater than the latest tag).
- Release notes are auto-generated by GitHub from merged PRs / commits since the previous tag. Keep PR titles descriptive — they end up in the release notes verbatim.

**How the workflow decides:**

1. Compute `pkg` from `package.json` and `latest` from `git tag --list 'v*' --sort=-v:refname | head -1`.
2. If `pkg > latest` (SemVer-sorted): release `v$pkg` (manual minor/major).
3. Otherwise: release `v<latest with patch incremented>`.
4. If the resulting tag already exists, skip (idempotent re-run protection).
