# pr-visual — agent notes

## Releases

Releases are automated by `.github/workflows/release.yml`. The workflow runs after the `CI` workflow finishes successfully on `master`, so a red CI run blocks the release. **A release is cut only when `package.json` `version` has been bumped above the latest `v*` git tag.** Merging a PR that does not touch `version` does not produce a release.

**Rules for AI agents:**

- **To ship a change, bump `package.json` `version` in the PR.** Pick the appropriate segment (patch / minor / major) for the change you're making. Also update the `version` fields in `.claude-plugin/plugin.json` and both locations in `.claude-plugin/marketplace.json` to the same value so all manifests stay aligned.
- **Merging without bumping does not release.** This is intentional: it lets you land refactors, docs, or chore work between releases without shipping.
- **Never create tags or GitHub Releases by hand.** The workflow owns `v*` tags.
- **Source of truth for the released version is the latest `v*` git tag.** The workflow reads `package.json` to decide whether to release; it never writes to the repo.
- Release notes are auto-generated by GitHub from merged PRs / commits since the previous tag. Keep PR titles descriptive — they end up in the release notes verbatim.
- `CHANGELOG.md` is maintained by hand. Add an entry in the same PR that bumps the version.

**How the workflow decides:**

1. Trigger: `workflow_run` after `CI` completes on `master`. If CI failed, the release job is skipped. `workflow_dispatch` is also available for manual re-runs.
2. Read `pkg` from `package.json` and `latest` from `git tag --list 'v*' --sort=-v:refname | head -1`.
3. If `pkg` is strictly greater than `latest` (SemVer-sorted) and `refs/tags/v$pkg` does not already exist: create the tag and publish the release.
4. Otherwise: do nothing.

**Checklist for a release PR:**

- [ ] Bump `version` in `package.json`.
- [ ] Bump `version` in `.claude-plugin/plugin.json` to the same value.
- [ ] Bump both `version` fields in `.claude-plugin/marketplace.json` to the same value.
- [ ] Add a `CHANGELOG.md` entry describing what's in the release.

## Commit messages — avoid `[skip ci]`

GitHub Actions matches `[skip ci]` (and `[ci skip]`, `[no ci]`, `[skip actions]`, `[actions skip]`) anywhere in a push's commit message — subject **or** body. A squash merge concatenates every commit on the branch into the merge commit body by default, so even a quoted or illustrative mention in a PR description or commit body will silently skip CI on `master`.

Rules:

- **Do not write `[skip ci]` or its aliases in commit messages or PR descriptions**, even in quotes, code fences, or "why we don't use this" explanations. Paraphrase instead (e.g., "the skip-ci token").
- **If you must reference it, use a Zero-Width-Space or similar break** (e.g., `skip ci` with a space) so the token no longer matches.
- A merge that silently skips CI also skips the release workflow, because Release runs via `workflow_run` on CI. The fix is a new PR whose merge triggers CI normally.
